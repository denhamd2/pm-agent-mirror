# XO Four-Stage Loop Metrics Flywheel - Design Note

**Status**: Deferred design (future plan). Captured 30 April 2026 during the `@xo-developer` ecosystem review (`/090-agent-improvement-advisor`). Not implemented.

**Owner**: PM, with implementation routed to `/workspace-audit` or `/value-metrics` skill.

## Problem

The `@xo-developer` four-stage parallel-review loop produces structured signal on every Tier 2 run (severity-tagged findings, triage verdicts, iteration counts, auto-apply vs escalation ratio, time-to-pass). Today, none of it is aggregated. The PM cannot answer:

- Which advisory behaviour fires most? (signals what to promote into the skill itself per the [retrospective process](../.cursor/agents/xo-developer-refs/advisory-playbook.md#playbook-retrospective-process))
- What is the auto-apply ratio vs escalation ratio? (signals whether the safety rails are calibrated correctly)
- How often does QA catch something the reviewer missed (and vice versa)? (signals whether one of the legs is theatre)
- What is the average iterations-to-pass for each Tier 2 mode? (signals which modes are stable vs which need rework)
- What is the offer count and capture rate for [Advisory #23](../.cursor/agents/xo-developer-refs/advisory-playbook.md#23-skill-capture-nudge-after-corrections)? (already partially instrumented via `_offers-log.md`; needs aggregation)

Without this telemetry, the playbook retrospective is qualitative and the system stays opaque.

## Proposed solution shape

A new metric set + dashboard, plugged into the existing `/workspace-audit` skill (which already produces `docs/pm-agent-scorecard.html` and `docs/pm-agent-architecture.html`). Not a new skill - extend the existing one.

### Data sources

1. **Advisory #23 offer log**: `.cursor/skills/xo-corrections/_offers-log.md` (already exists).
2. **Loop run log** (NEW): every four-stage loop run appends one line to a manifest file. Suggested location: `.cursor/skills/xo-builder/_loop-runs.md` (gitignored or kept short, append-only). Each line carries: timestamp, mode, modes-invoked-streams, reviewer-verdict, qa-verdict-or-NA, iterations, auto-apply count, escalation count, dismissed count, advisory-IDs cited.
3. **Capture files**: `.cursor/skills/xo-corrections/*.md` (count and freshness).
4. **Git log**: commits to `.cursor/agents/xo-developer-refs/advisory-playbook.md` for behaviour additions over time.

### Metrics

Tier 1 (loop health):
- **Pass rate per mode**: % of runs where reviewer = `approve` AND QA = `pass` on cycle 1.
- **Iteration distribution**: histogram of cycles to exit (1, 2, capped-at-2-with-findings-remaining).
- **Auto-apply ratio**: auto-applied findings / total findings, per mode.
- **Escalation ratio**: PM escalations / total findings, per mode.
- **Stream contribution**: % of findings from reviewer vs QA, per mode.
- **Cross-stream catch**: % of runs where one stream caught something the other missed (the high-value signal).

Tier 2 (advisory health):
- **Advisory fire rate**: how often each advisory ID is cited per 100 runs.
- **Top 3 firing advisories**: surface for promotion review.
- **Bottom 3 firing advisories**: surface for sunset review.
- **Advisory #23 specifically**: offers made, captures created, capture-to-offer ratio.

Tier 3 (PM behaviour):
- **PM escalation acceptance**: when escalated, does the PM pick A, B, or "explain more"?
- **HITL gate edits**: how often does the PM edit a Tier 2 diff vs straight `approve` / `reject`?

### Dashboard

Add a new tab to `docs/pm-agent-scorecard.html` (generated by `/workspace-audit`):

- Top: scorecard line for the loop (pass rate, average iterations, escalation rate).
- Middle: bar charts for fire-rate-per-advisory, sorted descending; top 3 highlighted for promote, bottom 3 highlighted for sunset.
- Bottom: stream-contribution pie chart, cross-stream catch counter, Advisory #23 offer/capture funnel.

Internal-only. Lives at localhost dashboard. Do **not** ship to GHE Pages (per `015-auto-commit-deploy.mdc`).

## Implementation phases (rough estimate)

1. **Phase 1 - Instrumentation**: extend [Advisory #17](../.cursor/agents/xo-developer-refs/advisory-playbook.md) Step 4 to require `@xo-developer` to append a one-line manifest entry to `.cursor/skills/xo-builder/_loop-runs.md` after every recap. ~1 hour.
2. **Phase 2 - Aggregation script**: Python script reading the manifest + offer log + capture files, producing a single JSON metrics blob. ~2 hours.
3. **Phase 3 - Dashboard tab**: Canvas Kit / React tab on the scorecard HTML reading the JSON. ~3 hours.
4. **Phase 4 - Retrospective integration**: extend the playbook retrospective process so it reads metrics output before deciding keep/sunset/promote. ~30 minutes.

Total: ~6-8 hours of focused work. Worth doing once the loop has accumulated 4-6 weeks of run data; running it with zero data is theatre.

## When to pick this up

- After 4-6 weeks of active four-stage loop use (so the manifest has signal).
- Triggered by a playbook retrospective that wants quantitative input.
- Triggered by the PM asking "is this loop actually catching things, or is it theatre?".

## Out of scope for this design

- Cross-tenant analytics (Pharos / Lightdash) - the loop is dev-SUV-only by design.
- Comparison with WATS coverage - WATS is a different verification surface; merging the metrics conflates two distinct quality signals.
- Real-time dashboards - batch refresh on `/workspace-audit` is sufficient.
