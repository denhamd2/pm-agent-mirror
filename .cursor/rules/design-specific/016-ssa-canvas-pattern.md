# SSA Canvas pattern

Use this rule together with [`015-sana-style-ui.md`](./015-sana-style-ui.md). 015 owns visual tokens and the SSA Title Strip / Starter Suggestions spec; **016 owns the behavioural contract**: how an SSA prototype boots, how it transitions from a generic home into a task canvas, and how the right pane is generated from an intent handler.

This rule is glob-scoped to `design/**/*` and React files. Zero token cost when you're not designing a Workday prototype.

## Intent

A **Self-Service Agent (SSA)** is a **generic, multi-capability agent surface** that can answer policy questions, run transactions, and coordinate multi-step workflows. The canonical SSA shape has two modes:

1. **Cold-start** — a single centred column on a white surface. Sparkle + greeting + starter suggestions + composer. No canvas. Users ask anything.
2. **In-task** — on any **canvas-triggering intent** (Create Offer, Create Job Req, Transfer Position, Backfill, etc.), the shell transitions into a **split-pane**: chat on the left (520px), canvas on the right (flex). Chat handles intent + rationale + light actions; canvas owns system-of-record state, validation, and the irreversible commit gate.

The canvas is **generated on the intent**. This is where Gen UI lives in Workday Recruiting: the right-pane layout and fields are a function of the agent's intent handler, not a hard-coded route. Prototypes express this as an intent registry; the shell supplies the layout.

## When to use the SSA Canvas pattern

| Use SSA Canvas when… | Don't use when… |
|---|---|
| The user's task is multi-step and **high-consequence** (regulated fields, legal copy, irreversible commits like sending an offer, opening a requisition, posting a transfer). | Single-field edits (e.g. change a start date on a candidate profile). Use the profile page. |
| The user benefits from seeing **system-of-record context next to the conversation** (cards, stepper, summary table). | Simple Q&A policy lookup. Use Ask Workday in the partial-panel chat. |
| NL disambiguation has real value (e.g. two Marias, two Portland reqs, ambiguous country). | Read-only dashboards. Use [`pm-agent-dashboard.tsx`](../../design/pm-agent-dashboard.tsx)-style data surfaces. |
| The flow ends in a **commit gate** that needs Human-in-the-Loop confirmation. | Candidate self-serve chat on a career site. Use the full-page conversational pattern in 015. |
| You need to **surface compliance warnings in context** (overlap rules, pay-band breaches, works-council, DE collective agreement, JP two-step, IN Aadhaar). | A modal task that already has a native Workday xoUi page. Prefer the native page; don't re-skin it as an SSA. |

If you're unsure, **ask the PM**: "Is this a *task* (SSA Canvas) or an *answer* (partial-panel chat)?". Don't ship an SSA Canvas for what should have been an embedded Ask Workday card.

## Mandatory pattern (behavioural contract)

Every SSA prototype **MUST**:

1. **Boot in cold-start mode** (`mode === 'cold-start'`). The default state is the generic SSA home — not a task-specific landing. Never open a prototype directly into a split-pane with a pre-filled task. The cold-start → in-task transition is the demo beat that proves the Gen UI value prop.

2. **Use `SsaShell` from `design/components/SsaShell.tsx`** for the outer chrome. Do not re-implement the top nav + title strip + layout math inline. Supply content via three slots:
   - `coldStart` — generic welcome + `SsaStarterSuggestions` + composer + AI disclosure.
   - `chat` — chat thread + composer + AI disclosure (split-pane left).
   - `canvas` — header card + stepper + active-step content (split-pane right).

   `SsaShell` owns top-nav + title-strip + split-pane layout. Add `WorkdayLeftTabBar` only when the brief explicitly calls for hub-style navigation around the SSA flow.

3. **Show a generic SSA cold-start** with:
   - A `SparkleMark` hero glyph (Sana link-accent blue).
   - A greeting in sentence case: *"Hi, I'm Self-Service Agent!"* (or your agent's real name).
   - A capability line: *"I can help answer general policy questions and help get things done."*
   - A 3–5 item `SsaStarterSuggestions` list (variant: `cold-start`). Include at least one canvas-triggering prompt and at least one out-of-scope prompt.
   - A `SanaCommComposer` with the placeholder *"Ask me anything."*
   - A neutral italic AI disclosure footer (*"This content was generated by AI. Review before use."*) — not a Canvas Kit `Banner`.

4. **Handle the canvas-trigger intent in the intent handler**. When the user sends a matching intent (e.g. `/\bcreate\s+an\s+offer\b/`, `/\bdraft\s+.*offer\b/`), the `AgentReply` must return `enterTaskMode: true` and the parent component flips `mode` to `'in-task'`. Chat history **survives the transition** — don't clear it; the user's first message and the agent's response both need to be visible in the new split-pane.

5. **Contextualise the first in-task agent response.** Don't jump straight to Step 1 with no explanation. The canonical first-response shape is:
   > *"Sure — I'll help you \[task\] for \[entity]. I've pre-filled \[fields\] from \[source\]. Tell me what to change, or work through the steps on the right."*
   
   This mirrors the Create Job Req demo (`ov-2700.png`) and anchors the split-pane's right-pane context.

6. **Render the right pane as a function of the intent.** Today this is done via per-step components in the prototype (`CandidateAndReqStep`, etc.). In Gen UI v2 (P2, deferred), this will be a typed registry `{ intent: SsaIntentId, renderCanvas: () => ReactNode }`. Either way, the canvas is the agent's output — not a hand-rolled wizard.

7. **Keep out-of-scope prompts honest.** When the user picks a cold-start suggestion that isn't wired up (e.g. "What is my time-off balance?"), reply with a three-part explainer: *"This prototype is scoped to the offer flow. Try 'Create an offer for a candidate' to see the Self-Service Agent open a task canvas on the right."* Don't pretend to answer.

## Gen UI canvas contract

The right pane is a **declared surface**, not a free-for-all. The agent's intent handler owns what goes there.

- **One intent → one canvas surface**. When `mode === 'in-task'`, the canvas is built for a specific intent (Create Offer, Create Job Req, Transfer Position). Intents that don't open the canvas must stay chat-only.
- **Chat narrates; canvas commits.** Chat messages are advisory — they explain what changed, why, and what's next. The canvas is where the user tick-boxes through validation, reviews evidence, and confirms the Send. If the user can commit from chat, you've built the wrong surface.
- **Chat changes MUST write through to the canvas.** If the user says "switch country to Germany" in chat, the corresponding cards on the right pane update immediately — and non-current steps that changed get a "review" dot on the stepper (318 finding from the Create Offer prototype).
- **Citations and evidence stay on the right.** Every AI-generated summary in the canvas (e.g. an approver packet) exposes link-out citations to the source data. Never claim something without a citation the user can open.
- **Human-in-the-Loop on commit.** The final step is always a review-and-send card with a pre-flight summary. No intent handler may call `onSend` directly from chat.

## Chat thread anatomy

SSA prototypes support two chat treatments. Pick one deliberately in the brief and keep it consistent through the flow:

- **Bubble mode** — appropriate for surfaces that follow the reusable communication-panel style (for example Create Offer). Use `SanaCommMessageBubble` for assistant and user messages, with Canvas Kit avatar treatment only when the reference requires it.
- **Summit plain-agent mode** — appropriate for Innovation Summit / Create Job Req references. Render agent turns as plain left-aligned text (no bubble, no avatar row), render user turns as right-aligned grey pills, and place feedback / grounding actions below agent turns: helpful, not helpful, `Sources`, optional `Show Steps`. The cold-start greeting may hide this feedback chrome so the hero moment stays plain text only.

For both treatments, `SanaCommComposer` stays canonical. Use `+ Sources` beneath the composer when the reference or brief includes source grounding, and keep the neutral italic AI disclosure at the bottom of the pane.

## Canonical components (from `design/components/SsaShell.tsx`)

Always import these; never re-implement inline:

- **`SsaShell`** — the layout wrapper. Takes `mode`, `tenantLabel`, `agentName`, title-strip handlers, and three slots (`coldStart`, `chat`, `canvas`). Handles `WorkdayTopNav` + `SsaTitleStrip` + height math + split-pane sizing.
- **`SsaTitleStrip`** — slim 48px header (toggle, agent name + caret, minimise, close). See 015 for tokens + forbidden elements.
- **`SsaStarterSuggestions`** — `↳` prompt list with rotate affordance. Variants: `'cold-start'` (max 560px, flush-left) vs `'in-task'` (max 380px, indented).
- **`SparkleMark`** — 4-point sparkle hero glyph used above the cold-start greeting.
- **`WorkdayLeftTabBar`** — optional outer-shell navigation for hub-style recruiter surfaces. Compose around `SsaShell` only when required by the brief/reference.

Shell tokens you may need:
- `SSA_CHAT_PANE_WIDTH_PX` (520)
- `SSA_COLD_START_MAX_WIDTH_PX` (720)
- `SSA_TITLE_STRIP_HEIGHT_PX` (48)

## Canonical copy (sentence case, British English, no emojis)

- **Cold-start greeting**: *"Hi, I'm Self-Service Agent!"*
- **Cold-start capability line**: *"I can help answer general policy questions and help get things done."*
- **Cold-start sub-prompt**: *"Explore suggestions below or ask me anything."*
- **Cold-start composer placeholder**: *"Ask me anything."*
- **In-task composer placeholder**: *"What would you like to do?"*
- **AI disclosure (both modes)**: *"This content was generated by AI. Review before use."* — neutral italic line, **not** a Canvas Kit `Banner`.
- **First in-task agent response opener**: *"Sure — I'll help you \[task\] for \[entity]."*
- **Out-of-scope reply**: three-part (problem / reason / next step), per 319 AI error shape.

## Forbidden in SSA prototypes

- **Skipping cold-start and booting into a split-pane.** Every SSA prototype starts on the generic home.
- **Advertising capabilities the agent doesn't have.** Starter suggestions must match what the intent handler actually understands (317-319 editorial rule).
- **Canvas Kit `Banner` for AI disclosure, agent identity, or starter-suggestion chrome.** Use neutral italic `BodyText` — see 015 `Banner` policy.
- **Hard-coded right-pane routing by URL slug.** The canvas is chosen by the agent's intent handler, not by a route. Keep the surface single-page.
- **Chat-only commit paths.** `onSend` (or equivalent irreversible action) lives on the canvas's review step, never inside an agent reply.
- **Breadcrumbs in an SSA prototype.** Inherits the 015 hard rule.
- **Clearing chat history on the transition.** The user's first message stays visible.

## 315 + 320 integration (how this rule participates in E2E)

- **315 (design brief)**: briefs for SSA flows must include an SSA section with (a) cold-start copy + starter set, (b) canvas-triggering intents, (c) per-intent canvas surface, (d) explicit out-of-scope prompts the agent will decline.
- **320 (prototype-developer)**: SSA prototypes must consume `SsaShell` / `SsaTitleStrip` / `SsaStarterSuggestions` / `SparkleMark` from `design/components/`. Inline re-implementation is a redesign signal.
- **318 (peer review)**: checks that chat changes write through to the canvas (review-dot on stepper), that the commit gate is canvas-side, and that cold-start copy matches the brief.
- **319 (copy review)**: applies the canonical copy above, the three-part out-of-scope shape, and verifies starter suggestions match intent-handler capability.
- **321 (visual review)**: validates cold-start layout (centred, 720px), split-pane (520px chat + flex canvas), title strip height (48px), and AI disclosure treatment.
